# Golden Gmail-web compose skeletons

These `golden_*.json` files are **sanitized structural skeletons** of real
messages composed in the Gmail **web** UI, used to assert that this server's
reply/forward builders produce byte-equivalent structure (the "indistinguishable
from Gmail-web" goal).

## Provenance (how these were captured)

- **Source:** messages *received* from third parties (`-from:me`) — i.e. composed
  in Gmail web by other people and sent to the account. This guarantees they are
  (a) genuinely Gmail-web output, (b) **not** generated by this MCP server (no
  circularity), and (c) current (recent `newer_than` window).
- **Provenance gate:** a sample counts only if its `Message-ID` ends with
  `@mail.gmail.com` **and** its HTML contains `gmail_quote` (the Gmail-compose
  scaffolding). Multiple samples per surface had to **agree** on the structural
  signature (MIME `multipart/alternative` shape + scaffolding probes) before the
  skeleton was accepted.
- **PII:** captured via the committed PII-safe `tools/golden_skeleton.extract_skeleton`.
  All addresses, display names, subjects, dates, and body text are redacted to
  placeholders; the variable quoted-original markup is dropped. Only structural
  scaffolding remains. No personal content is stored here.

Re-capture (local, not committed) with a script using `extract_skeleton` over
`-from:me subject:Re:/Fwd:` mail; bump the `newer_than` window if Gmail's markup
drifts.

## Key findings locked by these fixtures (captured 2026-06-24)

- **Reply** = `multipart/alternative` (text/plain + text/html). HTML reply trail:
  `<div class="gmail_quote gmail_quote_container">` → `<div dir="ltr" class="gmail_attr">On … wrote:<br></div>` → `<blockquote class="gmail_quote" style="…">{orig}</blockquote>`.
- **Reply blockquote style** (6/6 current samples agree): the **shorthand**
  `margin:0px 0px 0px 0.8ex;border-left:1px solid rgb(204,204,204);padding-left:1ex`
  — which the code's `gmail_web_mime.BLOCKQUOTE_STYLE` already matches. (An older
  pre-2025 sample showed a longhand `border-left-width/style/color` variant; the
  current agreeing majority is shorthand, so no change is needed — the regression
  test just locks it.)
- **Forward** = `multipart/alternative` alone, or `multipart/mixed` →
  [`multipart/alternative`] + attachment part(s) (`Content-Disposition: attachment`,
  base64) when attachments are carried. The forwarded content sits directly inside
  the `gmail_quote_container` after the `gmail_attr` header — **no `blockquote`**,
  and the plain-text forwarded block is **not** `> `-quoted. The forward header
  uses `<strong class="gmail_sendername" dir="auto">{name}</strong>` for the From
  line (current Gmail; older samples lacked `gmail_sendername`).
- **Content-Transfer-Encoding is content/part-dependent:** quoted-printable for
  mostly-ASCII parts, base64 for heavily-non-ASCII parts — and it can differ
  *per part* within one message (e.g. a forward with a base64 text/plain and a
  quoted-printable text/html). Handled by the `choose_cte` heuristic task.
- **Boundary** matches `^0{12}[0-9a-f]{16}$` (the `gmail_boundary()` generator).

## Files
- `golden_reply.json` — reply, no attachments.
- `golden_reply_attach.json` — reply with attachment (multipart/mixed).
- `golden_forward.json` — forward, no attachments.
- `golden_forward_attach.json` — forward with attachment (multipart/mixed).
